# ArchiAI → 10× Roadmap

**A plan to turn the current concept-layout MVP into a tool that stands out, grounded in the real `archiai-saas` codebase, reverse-engineered from Hypar, with Scrapling-powered learning and a crafted UI.**

---

## 0. Read this first — what you actually have today

Before any "10x" talk, here is the honest state of the repo, because the plan only makes sense against it.

**Backend (`backend/app`)**
- A genuinely decent **deterministic NLP pipeline** in `services/parser/` (`normaliser → room_extractor → size_resolver → constraint_extractor → building_inference → merger`) plus controlled vocabularies in `parser/data/`.
- A **layout engine** in `services/layout_service.py` (~1,500 lines after Sprint 16) that:
  - groups rooms into zones (`front_public_flow`, `public`, `semi_private`, `private`, `service`, `circulation`),
  - places them via a **tiled, space-filling algorithm** (`_tile_rooms`) for all building types as of Sprint 16 (residential + commercial), with `_place_rooms`/row-banding kept only as a fallback,
  - threads parser adjacency constraints (MUST/SHOULD) into row ordering,
  - runs **overlap repair** (`_repair_rooms`), generates **boundary + partition walls** (`_architectural_markers`, `_generate_partition_walls`), adds **stairs**, and
  - builds candidates and keeps the highest `score`.
- A **quality scorer** (`layout_quality_service.py`) that penalises overlaps, missing rooms, and bad adjacencies.
- A **Vastu** module (`parser/vastu.py`).
- A **scraper** (`services/scraper_service.py`) that is just `httpx` + `html.parser` visible-text extraction + `robots_txt_checker`, feeding a keyword cleaner (`scraper_cleaning_service.py`) into sparse `LayoutPattern` rows that nudge sizing/zoning/adjacency.

**Frontend (`frontend/src`)**
- React 18 + Vite + R3F. Rooms render as **boxes with edge lines** (`canvas/RoomMesh.tsx`), floating HTML labels via `drei`, drag-on-plane editing, view modes `3d / top / floor_plan`.
- An **Inspector** (`canvas/Inspector.tsx`) already shows editable **W / D / H** in a side panel.
- Versioning, auto-save, workspaces, sharing, PNG/PDF export all exist.

**The blunt diagnosis.** The skeleton is good. The reasons it does *not* yet stand out:
1. Even after Sprint 16 unifying placement onto the tiler, there is still no real **circulation network**, **no doors between rooms**, and **no plot/site** concept.
2. Everything is **inferred from prose**; there are almost no **explicit parametric controls** (plot size, setbacks, orientation, FAR, target areas).
3. The scraper learns **almost nothing structural** — it pulls visible text and keyword-matches it.
4. The UI is **functional Tailwind admin-panel** styling, not a crafted product.
5. Dimensions live in a side panel, not **on the model** where the user is looking.

Everything below attacks those five gaps.

## 1. The 10× thesis (what "stand out" means here)

Most "AI floor plan" apps do one of two things: (a) generate a *picture* of a plan with a diffusion model (pretty, dimensionally meaningless, uneditable), or (b) drag-and-drop room boxes (manual, no intelligence).

**Hypar's whole insight is that neither is interesting** — the value is a *parametric, data-driven, rule-composed* engine that produces **dimensionally real, editable, metric-scored** buildings, fast, with many options.

So ArchiAI's bet, in one sentence:

> **A parametric generative floor-plan engine that turns a brief *and* a plot into many dimensionally-real, code-aware, editable layouts — with priors learned from real architecture — wrapped in a UI a non-architect can use in 60 seconds.**

The plumbing for this already exists in good part. The 10× is in five pillars:

| Pillar | One-line goal | Maps to user's ask |
|---|---|---|
| **A. Planning engine** | Real envelope + space-partition + circulation + openings | "lot of logic and improvement in planning a house structure" |
| **B. Dimensions & interaction** | Dimensions, area, clearances shown *on the block* on click | "uses dimensions… shows them directly when clicking a block" |
| **C. Data learning** | Scrapling → structured patterns → statistical priors → engine | "use scrapling… ArchDaily/Dezeen… learn from scraped data" |
| **D. UI/UX** | Minimal, stylish, one-screen, option gallery | "minimalistic yet stylish ui which stands out" |
| **E. Interop & hardening** | SVG/DXF/glTF/IFC export, prod build, auth | (Hypar parity + investor-grade polish) |

## 2. Hypar, reverse-engineered into parts you can actually build

### 2.1 The "Function" model (the core idea worth stealing)
**Hypar:** every capability (core placement, façade, structure, space planning, daylight) is an independent **Function** with typed inputs/outputs, composed into a **workflow**. Re-running with new params regenerates everything downstream.

**ArchiAI build:** refactor `layout_service.generate_layout` from one large function into a **pipeline of pure functions** over a shared `BuildingModel`:

```
brief + params
   → EnvelopeFn        (plot, setbacks, FAR → buildable rectangle/L/U)
   → ProgramFn         (rooms + target areas + adjacency graph)   [mostly exists]
   → ZoningFn          (assign rooms to zones & floors)            [exists]
   → PartitionFn       (space-partition the envelope into rooms)   [NEW — the big one]
   → CirculationFn     (corridors + door graph)                    [NEW]
   → OpeningsFn        (doors on shared walls, windows by orient)  [partial today]
   → StructureFn       (columns/grid — optional, later)            [NEW, optional]
   → FurnitureFn       (suggested furniture per room — optional)   [NEW, optional]
   → MetricsFn         (areas, ratios, daylight proxy, score)      [extend quality svc]
```

Each `Fn` takes `BuildingModel` + params, returns a new `BuildingModel`. This is **the single most important refactor** — it makes options, learning, and new features additive instead of surgery on a monolith.

### 2.2 Parametric inputs (stop inferring everything)
Add an explicit **`DesignParams`** object alongside the prompt:
- `plot`: `{ width_m, depth_m, shape }` (or "draw your plot")
- `setbacks`: front/rear/side
- `far` / max coverage, `floors`, `floor_height`
- `orientation` (which edge faces the road / north)
- per-room `target_area` overrides
- toggles: `vastu`, `open_plan`, `corridor_style`

Prompt parsing still works (great for first draft), but params become first-class and editable.

### 2.3 Optioneering / generative engine
You already have mini-optioneering (multiple candidates, keep best `score`). Generalise it:
- Vary **seed parameters** (zone order, partition split ratios, corridor placement, aspect-ratio bias) to produce **6–12 candidates**.
- Score each with the extended `MetricsFn`.
- Return an **option set** to the UI as a **gallery** (thumbnails + key metrics), not a single answer. "Generate more" reseeds.

### 2.4 Real circulation (the thing your plans are missing most)
See **Pillar A**. This is the difference between "boxes near each other" and "a house you can walk through."

### 2.5 3D web preview + live metrics
Add a **metrics HUD** (total area, carpet area, efficiency %, room count, daylight proxy) and the **dimension overlay** (Pillar B).

### 2.6 Suggestions UX
When a user renames/retypes a room in the Inspector, **re-run `FurnitureFn`/`PartitionFn` locally** for that room and offer "apply suggestion." Cheap, deterministic, feels magical.

### 2.7 Interoperability / export
Add **SVG floor plan** (vector, dimensioned), **DXF** (opens in any CAD), **glTF** (3D, via three.js exporter), and later **IFC-lite**. See Pillar E.

### 2.8 What to deliberately skip (don't drown)
- Structural steel / MEP / duct routing — irrelevant to a concept tool for homeowners/students. Skip for ≥12 months.
- Full IFC/Revit round-trip — IFC-*lite* export only, much later.
- A C#-style function marketplace — not your model; Python/TS stays the stack.

## 3. Pillar A — The planning-logic overhaul (the heart of the 10×)

Goal: a layout that **fills a real building envelope with no nonsensical gaps, has corridors and doors, and respects adjacency, daylight, and dimensions.** Replace "park rooms in rows/tiles" with "partition a footprint."

### 3.1 Step 1 — Compute a real envelope
New `EnvelopeFn`. Inputs: `plot` + `setbacks` + `far`/coverage (or, if no plot given, derive a sensible rectangle from total program area × an efficiency factor). Output: a `footprint` rectangle (later: L/U/courtyard shapes) per floor. Repurpose the existing footprint helpers (`_footprint_for_rooms`, `_shared_footprint`) as **the constraint**, not the byproduct.

### 3.2 Step 2 — Partition the envelope (the new core algorithm)
Replace row/tile packing with a **slicing-tree / Binary Space Partition (BSP)** allocator:

1. Build a **program tree**: zones → rooms, each with a **target area** and an **aspect-ratio preference** (squarish for bedrooms, wide for living, narrow-ok for service).
2. **Recursively split** the footprint rectangle. At each node, split along the longer axis; the **split ratio = (area of left subtree) / (total area of node)**. Choose split orientation to keep child aspect ratios near each room's preference.
3. Order children so that **MUST-adjacent rooms** (from `AdjacencyConstraint`, already threaded into the tiler as of Sprint 16) land in **sibling** cells, and **avoid pairs** land in different branches.
4. When a leaf's aspect ratio is too extreme, **carve a corridor strip** instead of a room (feeds Step 3).

This guarantees no gaps, no overlaps by construction, and dimensionally meaningful rooms.

Minimal shape of the new module (`services/planning/partition.py`):

```python
@dataclass
class Cell:
    x: float; z: float; w: float; d: float          # rectangle in metres
    room: RoomSpec | None = None                     # leaf holds a room
    children: list["Cell"] = field(default_factory=list)

def partition(footprint: Cell, program: ProgramNode) -> Cell:
    """Slicing-tree allocator: split footprint by area weights,
    biased to honour adjacency order and aspect-ratio prefs."""
    if program.is_leaf:
        footprint.room = program.room
        return footprint
    left, right = program.balanced_split()          # keeps MUST pairs together
    ratio = left.area / program.area
    a, b = footprint.split(axis=footprint.long_axis(), ratio=ratio)
    footprint.children = [partition(a, left), partition(b, right)]
    return footprint
```

Keep the **tiled algorithm as a fallback** for studios/tiny programs; run BSP for `house/apartment/villa/...`. Score both, keep the better — the candidate-selection harness in `generate_layout` already supports this pattern.

### 3.3 Step 3 — Circulation graph + corridors
New `CirculationFn` (`services/planning/circulation.py`):
- Build an **adjacency graph** of placed cells (nodes = rooms, edges = shared wall segments ≥ door width).
- Ensure every room is **reachable from the entry** through public space. If a private room only touches other private rooms, **insert a corridor cell** or widen an existing one.
- Result: a real **path from front door → hall/corridor → each room**.

### 3.4 Step 4 — Doors and windows that mean something
Extend `_architectural_markers` / `_generate_partition_walls`:
- **Doors:** place a door **opening** on the shared wall of each graph edge used by circulation (not just partition walls everywhere). Honour a **clearance rectangle** in front of each door and flag overlaps in the scorer.
- **Windows:** place on **exterior** walls, sized by room area, biased to the **daylight-favourable orientation** from `DesignParams.orientation`. Add a cheap **daylight proxy** metric = (exterior-wall length with windows) / (room area).

### 3.5 Step 5 — Feed it back into the scorer
Extend `layout_quality_service` with: reachability (every room connected), corridor efficiency, daylight proxy, door-clearance violations, aspect-ratio sanity. These metrics also become the **gallery filters** in the UI and the objective function for an optional optimiser.

**Net effect of Pillar A:** the output stops looking like coloured boxes and starts looking like a plan an architect would sketch — filling a plot, with a front door, a hall, rooms off the hall, and windows on the outside.

## 4. Pillar B — Dimensions & interaction

### 4.1 Dimensions rendered on the model
Add `frontend/src/components/canvas/DimensionAnnotations.tsx`, rendered for the **selected** room (and optionally on hover). Use R3F + `drei`'s `<Line>` and `<Html>`:
- Two **dimension lines** offset just outside the room footprint — one along **W**, one along **D** — with end ticks, labelled e.g. `4.20 m`.
- A centre **area badge**: `3.6 × 4.2 m · 15.1 m²` (and `sq ft` toggle).
- Lines/labels **update live** during drag/resize (subscribe to the same `useCanvasStore` room object `RoomMesh` already uses).

### 4.2 Make it feel like CAD-lite
- **Global "Show dimensions" toggle** in `EditorToolbar.tsx`.
- **Snapping** + a **dimension grid** (the store already has `snapToGrid`) so dragging shows live distance to neighbours.
- **Clearance overlay** for doors (from Pillar A) drawn as a translucent rectangle.
- **Metrics HUD** (top-right): total built-up area, carpet area, efficiency %, room count, daylight score — updates live.

### 4.3 Click → contextual info
On select, the Inspector header shows a compact **room card**: type, area, % of total, adjacency ("opens to: Hall, Balcony"), and any **scorer warnings for that room**.

## 5. Pillar C — Scrapling + learning from real architecture

### 5.1 Do this responsibly (non-negotiable)
- **Respect `robots.txt` and each site's Terms of Service.** The existing `robots_txt_checker.py` and "permitted public-text references" posture stays. Scrapling supports `robots_txt_obey = True` on spiders.
- **Rate-limit**, identify the agent, cache aggressively.
- **Learn aggregate patterns, do not republish content.** Store *derived statistics* (typical areas, ratios, adjacency frequencies, program mixes, plan-type tags) — **not** copied article text or images.
- Prefer **official/open sources where they exist** (open architecture datasets, government building-code tables, Creative-Commons plan sets); treat ArchDaily/Dezeen as one signal among several.
- Attribute sources in `ScrapedRecord.source_url`.

### 5.2 Swap the fetcher to Scrapling
Keep the existing `fetch_public_page(url)` interface; change its body. Install: `pip install "scrapling[fetchers]"` then `scrapling install`. Needs Python 3.11+ and pulls browser binaries — consider a dedicated `scraper` container.

```python
from scrapling.fetchers import StealthyFetcher, Fetcher
StealthyFetcher.adaptive = True

async def fetch_public_page(url: str) -> "Page":
    page = await Fetcher.async_fetch(url, timeout=15)
    if page.status >= 400 or _looks_blocked(page):
        page = StealthyFetcher.fetch(url, headless=True, network_idle=True)
    return page
```

Gets Cloudflare/anti-bot bypass, JS rendering, and adaptive selectors that self-heal when source sites change markup.

### 5.3 Crawl with a Spider, extract structure not text
Add `services/scraping/spiders/`. For each project page extract: building type/program, total area, room list with per-room areas where stated, plan descriptors ("open plan", "central courtyard", "L-shaped"), adjacency language. Store in a new `ScrapedProject` table.

### 5.4 Turn many records into priors
New `services/learning/pattern_aggregation.py` rolls `ScrapedProject` rows into priors per building type: area distributions, room-to-total-area ratios, adjacency probabilities, program-mix templates, layout-type frequency. These plug into the existing `LayoutPatternRules` abstraction the engine already consumes.

### 5.5 Optional ML layer (only after priors work)
An adjacency-prediction model and a learned realism scorer, both deterministic at inference, trained offline against scraped-vs-randomised layouts. Keep every prior traceable to source rows (extend the existing `appliedPatternCount`/`patternDataSource` metadata).

## 6. Pillar D — UI/UX: minimal, stylish, unmistakably yours

### 6.1 Design language
- **One screen, three zones:** left = brief + params (collapsible), centre = 3D canvas + option gallery strip, right = Inspector + metrics.
- **Palette:** near-white canvas, ink text, one confident accent; desaturate the current `ROOM_COLORS` ~20% for an "architectural drawing" feel.
- **Type:** one humanist sans, a real scale, tabular figures for dimensions/metrics.
- **Space + restraint:** more whitespace, hairlines instead of boxes, soft shadows only on floating panels.
- **Motion:** 150–200 ms ease on selection/panel-open/option-switch; camera eases between view modes instead of cutting.

### 6.2 Signature interactions
- **Option gallery:** generated candidates as a horizontal strip of mini-thumbnails with their top metric; click to load, "reseed" for more.
- **Dimensions-on-click** (Pillar B).
- **Command palette** (⌘K): "add bedroom", "make open plan", "rotate plan 90°", "export DXF".
- **Empty state that teaches:** pre-filled example prompt + one-tap presets.
- **Plan view = real drawing:** walls as solid poché, doors as arc swings, dimension strings outside — not just top-down boxes.

### 6.3 Onboarding & trust
- A 20-second first-run flow.
- An "insights" popover citing which rules/priors fired and from which sources — transparency as a feature.

## 9. How not to look like every other "AI floor plan" app

- **Dimensional truth.** Competitors output pictures; you output metres, areas, and editable geometry.
- **Plot-aware.** Let a homeowner say "my plot is 30×40 ft, road on the south" and respect it.
- **Explainable priors.** "We learned typical room sizes from real built projects, and here's which rules fired."
- **Regional intelligence.** Pair the existing Vastu module with India-specific program priors (pooja room, utility, balcony norms) from scraped data — a defensible niche bigger players ignore.
- **60-second usable.** A non-architect gets a real, dimensioned plan in under a minute.
